JavaScript：欺骗词法作用域之with

2017-11-07   #with 

读书笔记《你不知道的JavaScript（上卷）》词法作用域。

欺骗词法作用域会导致性能下降，不推荐使用。

但仍然要了解下原理。

自问自答

  • 问：width是怎么欺骗（修改）词法作用域的呢？
  • 答：width可以接受一个对象作为参数，并凭空创建了一个全新的词法作用域，但这个块内部正常的var声明并不会被限制在这个块的作用域中，而是被添加到with所处的函数作用域中。

实例分析

function foo(obj) {
    width (obj) {
        a = 2;
    }
}
var o1 = {
    a: 3
};
foo(o1);
console.log(o1.a); // 2

栈分析：

全局作用域 销毁
console作用域销毁 数据结果 2 console作用域 创建
调用console.log(o1.a)
foo作用域 销毁
with作用域 销毁 改变o1.a = 2 with作用域 创建
调用with(obj) obj = o1 foo作用域 创建
调用foo(o1) o1 = { a: 3 } 全局作用域 创建

做一点改变

function foo(obj) {
    width (obj) {
        var a = 2;
    }
}
var o1 = {
    a: 3
};
foo(o1);
console.log(o1.a); // 2

// 上面代码和下面代码功能相同，下面代码更便于分析，执行完with之后

function foo(obj) {
    var a;
    width (obj) {
        a = 2;
    }
}
var o1 = {
    a: 3
};
foo(o1);
console.log(o1.a); // 2

栈分析：

全局作用域 销毁
console作用域销毁 数据结果 2 console作用域 创建
调用console.log(o1.a)
foo作用域 销毁
with作用域 销毁 改变o1.a = 2 var声明的变量被添加到函数foo中 with作用域 创建
var a（注意这里是后面添加的） 调用with(obj) obj = o1 foo作用域 创建
调用foo(o1) o1 = { a: 3 } 全局作用域 创建

再做一点改变

function foo(obj) {
    width (obj) {
        b = 8;
    }
}
var o1 = {
    a: 3
};
foo(o1);
console.log(o1.b); // undefined
console.log(b); // 8 不好！b被泄漏到全局作用域中了!

// 上面代码和下面代码功能相同，下面代码更便于分析，执行完with之后

function foo(obj) {
    width (obj) {
        b = 8;
    }
}
var b = 2; // b被泄漏到了全局作用域中
var o1 = {
    a: 3
};
foo(o1);
console.log(o1.b); // undefined
console.log(b); // 8 不好！b被泄漏到全局作用域中了!

栈分析：

全局作用域 销毁
console作用域销毁 数据结果 8 console作用域 创建
调用console.log(b)
console作用域销毁 数据结果 undefined console作用域 创建
调用console.log(o1.b)
foo作用域 销毁
with作用域 销毁 o1中没有找到b，而且b没有用var定义，根据非严格模式下LHS查询，自动创建一个全局变量b with作用域 创建
调用with(obj) obj = o1 foo作用域 创建
b = 8（注意这里是后面添加的） 调用foo(o1) o1 = { a: 3 } 全局作用域 创建

总结

  • width块内部正常的var声明并不会被限制在这个块的作用域中，而是被添加到width所处的函数作用域中。
  • width块内部没有用var声明的变量如果在参数对象中找不到会被泄漏到全局作用域中，如果能找到，只有声明被添加到with所处的函数作用域中。
  • 严格模式下with被完全禁止。
  • 严格模式下，间接或非安全地使用eval也被禁止。

  • 不推荐使用eval或with欺骗词法作用域，它们带来的好处远远抵不上带来的损耗。引用书中解释。

    如果引擎在代码中发现了 eval(..) 或 with，它只能简单地假设关于标识符位置的判断都是无效的，因为无法在词法分析阶段明确知道 eval(..)会接收到什么代码，这些代码会如何对作用域进行修改，也无法知道传递给with用来创建新词法作用域的对象的内容到底是什么。

    最悲观的情况是如果出现了 eval(..) 或with，所有的优化可能都是无意义的，因此最简单的做法就是完全不做任何优化。
